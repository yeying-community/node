01. 应用入驻流程
Step 1: 钱包登录 Node 应用，进入“应用中心”。
Step 2: 在“发布应用”填写信息并保存，系统会创建应用记录。
Step 3: 创建成功后会得到应用 AppId（即 applications.uid）。
Step 4: 提交上架申请并完成审批。
Step 5: 上架成功后 is_online=true，应用进入可用状态。
Step 6: 第三方应用把 AppId 作为 clientId 接入中心化 UCAN。
关键点： clientId 不在发布时新生成，使用的是创建应用时的 AppId(uid)。应用必须已上架，且需在发布页配置 redirectUris 回调白名单。
02. UCAN 授权流程（中心化 Issuer）
1) 业务端发起授权请求：提交 address + clientId + redirectUri。
2) Node 校验 clientId + redirectUri：仅接受已上架应用 AppId，且 redirectUri 必须命中应用发布字段 redirectUris。
3) 用户在 Node 验证页输入 TOTP，授权通过后拿到一次性 code。
4) 业务端用 code + clientId + redirectUri 兑换 JWT + UCAN。
5) 业务端携带 Bearer Token 调用受保护接口，服务端按统一 UCAN 规则验证。
推荐实践： 无钱包插件时优先走此流程；有钱包插件时可继续钱包直连，两者服务端验证逻辑一致。
03. 参数与约束
  • clientId：必须使用应用市场 AppId（applications.uid）。
  • redirectUri：必须命中该应用发布字段 redirectUris。
  • audience：需与服务端 UCAN audience 配置一致。
  • capability.with/can：必须与服务端能力策略一致。
04. 常见报错处理
  • redirectUri is not allowed：回调地址未进入允许列表。
  • Unauthorized clientId：clientId 不存在或应用未上架。
  • UCAN audience mismatch：Token 的 aud 与服务端期望不一致。
  • UCAN capability denied：签发能力不足，需调整能力并重签。
  • USER_ROLE_DENIED：账号角色不满足发布/审批权限。

06. 快速开始（5 分钟）
  1. 钱包登录并进入“我的配置”。
  2. 刷新 TOTP 状态，确认服务已就绪。
  3. 加载认证器配置并扫码绑定 TOTP。
  4. 填写 address/clientId/redirectUri 并创建授权请求。
  5. 批准授权并兑换 code，验证 JWT/UCAN 可用性。
07. 排查建议
  • 先查服务端 logs/，再看浏览器控制台与 Network 面板。
  • 优先核对三元组：address、clientId、redirectUri。
  • 涉及 UCAN 校验时，同时核对 audience 和 capability。
