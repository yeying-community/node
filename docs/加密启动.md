# 密钥初始化与加密启动

本文档说明 Node 服务在生产环境需要的核心密钥、初始化脚本，以及“本地加密存储 + 启动时输入密码解密”的落地方式。

## 1. 当前服务实际依赖的核心密钥

以下是代码中“会影响鉴权安全边界”的密钥材料：

- `JWT_SECRET`
  - 用途：SIWE 登录后的 JWT 签发与校验（`src/auth/siwe.ts`）。
  - 要求：必须配置、长度至少 32、不能使用默认占位值。
- `UCAN_ISSUER_PRIVATE_KEY`
  - 用途：中心化 UCAN 签发私钥（`src/auth/ucanIssuer.ts`）。
  - 生效条件：`ucanIssuer.enabled=true` 且 `mode=issue|hybrid`。
- `UCAN_ISSUER_DID`
  - 用途：中心化 Issuer 的 DID 标识；应与私钥推导结果一致。
  - 生效条件：与 `UCAN_ISSUER_PRIVATE_KEY` 同步使用。
- `TOTP_AUTH_TOTP_MASTER_KEY`
  - 用途：TOTP 随机密钥的加密主密钥（`src/auth/totpAuth.ts`）。
  - 生效条件：`totpAuth.enabled=true`。

补充：
- 运行中还会生成 `challenge/nonce/requestId/sessionToken` 等随机值，但这些是临时会话数据，不属于部署初始化固定密钥。

## 2. 初始化脚本（生成 + 加密落盘）

已提供脚本：

- `scripts/init-secrets.cjs`
  - 自动生成：
    - `JWT_SECRET`
    - `UCAN_ISSUER_PRIVATE_KEY`
    - `UCAN_ISSUER_DID`（由私钥推导）
    - `TOTP_AUTH_TOTP_MASTER_KEY`
  - 交互输入密码后，使用 `AES-256-GCM + PBKDF2` 加密保存。
  - 默认输出文件：`run/secrets.enc.json`（权限 `600`）。

示例：

```bash
node scripts/init-secrets.cjs
# 或指定文件
node scripts/init-secrets.cjs --file run/prod-secrets.enc.json
```

## 3. 启动时密码解密（避免明文出现在配置文件）

已接入 `scripts/starter.sh`：

- 当 `SECRETS_FILE` 指向的加密文件存在时，启动前会提示输入密码（或使用 `SECRETS_PASSWORD_FILE` / `NODE_SECRETS_PASSWORD`）。
- Node 进程启动后会在进程内解密密钥，并写入内存态密钥仓（不注入业务密钥环境变量）。
- 若密码错误或文件损坏，启动会失败并提示错误。

示例：

```bash
SECRETS_FILE=run/secrets.enc.json bash scripts/starter.sh start
```

也可重启：

```bash
SECRETS_FILE=run/secrets.enc.json bash scripts/starter.sh restart
```

本地开发（`npm run dev`）推荐：

```bash
npm run dev:secure
```

说明：
- `dev:secure` 会交互输入口令，并将口令仅传递给当前 Node 开发进程。
- 密文文件路径优先级：`--file` > `SECRETS_FILE` > `config.js` 的 `secrets.file` > 默认 `run/secrets.enc.json`。

## 4. 生产建议

- `config.js` 中不要保留真实密钥，尽量留空或占位，改由加密密钥文件 + 启动密码注入。
- 密钥文件（如 `run/secrets.enc.json`）应纳入主机级备份策略，但不能入库。
- 启动密码建议：
  - 长度 >= 16
  - 与业务账号密码分离
  - 定期轮换
- 若是无人值守场景，可通过 `NODE_SECRETS_PASSWORD` 注入密码，但安全性低于交互输入，应谨慎使用并配合主机密钥管理。

## 5. 无明文收敛策略（推荐落地）

- 单一密钥源：仅保留加密密钥文件 `run/secrets.enc.json` 作为运行密钥来源，`config.js` 中密钥字段保持空值。
- 启动前置校验：
  - `JWT_SECRET` 缺失时，服务直接启动失败。
  - 当 `ucanIssuer.enabled=true` 且 `mode=issue|hybrid` 时，若私钥/DID 不可用，服务直接启动失败。
  - 当 `totpAuth.enabled=true` 时，若 `TOTP_AUTH_TOTP_MASTER_KEY` 不可用，服务直接启动失败。
- 运行期不落盘明文：密钥仅在进程内存中驻留，不回写到配置文件。
  - 补充：在当前方案中，业务密钥默认不进入环境变量；仅口令可能通过 `NODE_SECRETS_PASSWORD`（可选）传递。

## 6. 常见运维问题

### 6.1 密钥文件丢失或误删怎么办

- 有备份：从备份恢复 `secrets.enc.json`，输入原密码即可恢复业务连续性。
- 无备份：只能执行重新初始化，影响如下：
  - `JWT_SECRET` 变化：所有历史登录态失效，需要重新登录。
  - `UCAN_ISSUER_PRIVATE_KEY` 变化：Issuer DID 变化，依赖旧 Issuer 的 token 需要重新授权获取。
  - `TOTP_AUTH_TOTP_MASTER_KEY` 变化：历史绑定的 TOTP 密文无法解密，需要用户重新绑定认证器。

### 6.2 是否支持多实例启动

- 支持。建议所有实例共享同一份密钥材料（同一加密文件内容，或同一密钥管理系统下发的等值环境变量）。
- 若实例间密钥不一致，会出现登录态、UCAN 签发/校验、TOTP 解密不一致，表现为随机鉴权失败。

### 6.3 是否只保留一份、并从配置文件移除私钥

- 建议是。生产环境应将 `config.js` 的以下字段置空并长期保持：
  - `auth.jwtSecret`
  - `ucanIssuer.privateKey`
  - `totpAuth.totpMasterKey`
- 这些值统一通过加密密钥文件（或密钥管理系统）注入，避免明文泄露和多份不一致。
