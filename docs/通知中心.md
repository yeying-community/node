# 通知中心

本文档定义 Node 作为“通知中心”时的职责边界、事件模型、通知出口与持续演进方向。

## 1. 目标定位

通知中心不是一个独立站点，而是 Node 枢纽平台中的一个核心域。它的职责是：

- 接收应用中心、授权中心、MPC 协调器产生的关键业务事件
- 将事件收敛成统一通知模型
- 按订阅规则分发到不同通知出口
- 为用户、应用、运维系统提供统一通知视图

一句话概括：

> Node 负责产生和管理“可信通知事件”，并提供统一分发能力。

## 2. 为什么要独立成域

当前系统已经存在多类“天然通知源”：

- 应用审核通过 / 驳回
- 应用上架 / 下架
- 中心化授权请求创建 / 待确认 / 已确认 / 已过期
- MPC 会话创建 / 成员加入 / 消息到达 / 签名完成 / 签名失败
- 系统异常 / 安全告警 / 配置变更

如果这些通知仍散落在各业务流程中，会有几个问题：

- 事件结构不统一，难以复用
- 前端需要分别对接多个业务接口
- 后续增加 Webhook、站内信、邮件、短信时需要重复开发
- 无法形成统一的通知审计与状态跟踪

因此通知中心应被视为一个独立域，而不是若干零散的“发送消息逻辑”。

## 3. 通知中心的职责边界

### 负责
- 定义通知事件标准模型
- 负责通知事件持久化与查询
- 负责通知订阅与通知出口编排
- 负责通知投递状态追踪
- 负责按身份与资源范围做通知可见性控制

### 不负责
- 不替代应用中心本身的业务状态机
- 不替代授权中心的鉴权逻辑
- 不替代 MPC 协调器的消息交换协议
- 不直接管理对象存储、邮件服务、短信服务的底层实现

通知中心是“控制面”，不是每种消息系统的“数据面”。

## 5. 通知事件模型

建议统一采用如下逻辑模型：

```ts
type NotificationEvent = {
  id: string
  type: string
  source: 'application' | 'audit' | 'auth' | 'ucan' | 'totp' | 'mpc' | 'system'
  subjectType: 'user' | 'application' | 'audit' | 'session' | 'system'
  subjectId: string
  actor?: string
  audienceType: 'user' | 'owner' | 'approver' | 'participant' | 'admin' | 'public'
  audienceIds?: string[]
  level: 'info' | 'success' | 'warning' | 'error'
  title: string
  body: string
  payload?: Record<string, unknown>
  createdAt: number
}
```

### 字段说明
- `type`：通知类型，供前后端做业务区分
- `source`：事件来源域
- `subjectType / subjectId`：通知所关联的核心业务对象
- `actor`：触发该通知的主体（钱包地址 / did / 系统）
- `audienceType / audienceIds`：通知面向谁
- `level`：通知严重级别
- `title / body`：面向产品展示的标准文本
- `payload`：结构化扩展数据

## 6. 通知出口

通知中心应统一支持多种出口，但每种出口都是同一事件模型的不同投递方式。

### 6.1 站内通知
- 面向已登录用户
- 在 Web UI 中展示未读/已读状态
- 适合审核结果、授权确认、MPC 状态变更

### 6.2 SSE / 实时推送
- 面向前端实时状态刷新
- 适合授权确认页、MPC 会话页面、审批工作台
- 当前系统已有 SSE / 事件流基础，可优先复用

### 6.3 Webhook
- 面向第三方应用或自动化系统
- 适合应用审核、授权成功、MPC 结果通知

### 6.4 邮件 / 短信 / IM（后续）
- 面向重要异步通知
- 例如审批完成、安全告警、重要授权确认

## 7. 通知状态模型

不同出口需要不同状态，但建议抽象成统一投递状态：

```ts
type NotificationDeliveryStatus =
  | 'pending'
  | 'delivered'
  | 'read'
  | 'failed'
  | 'expired'
```

### 说明
- `pending`：已生成，待投递
- `delivered`：已送达目标出口
- `read`：用户已读，仅对站内通知生效
- `failed`：投递失败，可重试
- `expired`：通知已过期，不再投递

## 8. 与现有系统的结合方式

### 8.1 与应用中心结合
- 应用发布、审核状态变化都先产出统一事件
- 通知中心负责决定是否展示为开发者通知、审核通知或外部回调

### 8.2 与授权中心结合
- TOTP 授权请求、中心化 UCAN 签发结果可直接产生通知事件
- 可复用于站内提示、实时确认页刷新、Webhook 回调

### 8.3 与 MPC 协调器结合
- MPC 本身已有事件流能力
- 通知中心不替代 MPC 消息流，而是提取其中“业务级关键事件”

### 8.4 与审计体系结合
- 所有重要通知事件应可追溯
- 通知产生与投递结果应能进入审计视图

## 9. 推荐的落地顺序

### 第一阶段：统一事件模型
- 先不急于增加外部通知渠道
- 先把应用审核、授权确认、MPC 关键事件统一成通知事件模型

### 第二阶段：站内通知 + SSE
- 为 Web 前端提供统一通知列表与实时事件订阅
- 优先覆盖审批台、我的配置、授权确认页、MPC 会话页

### 第三阶段：Webhook
- 为第三方应用和自动化系统提供事件订阅能力
- 支持按应用、事件类型、资源范围订阅

### 第四阶段：多出口能力
- 接入邮件、短信、IM 等外部通知渠道
- 引入通知模板、重试策略、死信队列

## 10. 持续演进建议

后续每次迭代通知中心时，优先补这几个方面：

- 新增事件类型时，先写入本文档
- 明确该事件的 `source / audience / payload` 结构
- 明确它属于“站内通知、实时通知、Webhook、外部消息”的哪一类
- 明确是否需要投递状态追踪与重试

建议把通知中心长期维护成 Node 枢纽平台的一个独立能力域，而不是前端临时提示的集合。

## 11. 数据结构草案

建议通知中心至少拆成三类表：

### 11.1 通知事件表 `notifications`

用于保存“平台级通知事件”的主记录。

```ts
type NotificationDO = {
  uid: string
  type: string
  source: string
  subjectType: string
  subjectId: string
  actor: string
  audienceType: string
  audienceIds: string
  level: string
  title: string
  body: string
  payload: string
  status: string
  createdAt: string
  updatedAt: string
  expiresAt: string
}
```

建议字段说明：
- `uid`：通知唯一标识
- `type`：通知类型，例如 `audit.approved`
- `source`：来源域，例如 `audit` / `mpc`
- `subjectType` / `subjectId`：关联对象
- `actor`：触发主体
- `audienceType`：受众类型
- `audienceIds`：受众标识列表，JSON 字符串
- `payload`：扩展上下文，JSON 字符串
- `status`：通知主状态，例如 `pending` / `delivered` / `expired`
- `expiresAt`：过期时间，可为空字符串

### 11.2 用户通知收件箱表 `notification_inboxes`

用于站内通知场景，按“用户视角”落地收件状态。

```ts
type NotificationInboxDO = {
  uid: string
  notificationUid: string
  recipient: string
  recipientType: string
  isRead: boolean
  readAt: string
  deliveredAt: string
  archivedAt: string
  createdAt: string
  updatedAt: string
}
```

说明：
- `notificationUid` 指向通知主表
- `recipient` 建议使用 did / address 的规范化值
- `isRead` / `readAt` 用于站内通知已读状态
- `archivedAt` 预留给归档能力

### 11.3 通知投递表 `notification_deliveries`

用于保存每个通知出口的投递记录。

```ts
type NotificationDeliveryDO = {
  uid: string
  notificationUid: string
  channel: string
  target: string
  status: string
  attemptCount: number
  lastError: string
  deliveredAt: string
  nextRetryAt: string
  createdAt: string
  updatedAt: string
}
```

说明：
- `channel`：例如 `inbox` / `sse` / `webhook` / `email`
- `target`：目标地址，例如 webhook URL、邮箱地址、用户标识
- `attemptCount`：投递次数
- `lastError`：最近一次错误
- `nextRetryAt`：失败后的下次重试时间

## 12. 接口现状与后续规划

### 12.0 一期已落地能力

当前代码已经落地以下最小能力：

- 表：
  - `notifications`
  - `notification_inboxes`
- 用户接口：
  - `GET /api/v1/public/notifications`
  - `GET /api/v1/public/notifications/unread-count`
  - `POST /api/v1/public/notifications/:uid/read`
  - `POST /api/v1/public/notifications/read-all`
  - `GET /api/v1/public/notifications/stream`
- 已接入事件：
  - `audit.created`
  - `audit.approved`
  - `audit.rejected`
  - `totp.request_approved`
  - `totp.request_expired`

其中审批通知当前会额外携带：

- `payload.auditId`
- `payload.targetUid`
- `payload.targetDid`
- `payload.targetVersion`

用于前端从通知直接跳转到具体应用详情/审批上下文，而不是只跳转到列表页。

当前 SSE 为单实例内存分发模型，适合本地开发和单实例部署；多实例一致性后续再接 Redis/统一事件总线。

通知中心建议优先暴露“用户收件箱接口 + 实时订阅接口 + 管理/回调接口”三类能力。

### 12.1 用户收件箱接口

#### 查询通知列表
```http
GET /api/v1/public/notifications
```

当前已支持查询参数：
- `page`
- `pageSize`
- `unreadOnly=true|false`

后续可扩展：
- `source=application|audit|auth|mpc|system`
- `level=info|success|warning|error`

当前返回结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "page": {
      "page": 1,
      "pageSize": 20,
      "total": 0
    }
  }
}
```

#### 标记已读
```http
POST /api/v1/public/notifications/:uid/read
```

### 12.2 实时订阅接口

#### SSE 订阅
```http
GET /api/v1/public/notifications/stream
```

当前行为：
- 建连后先返回 `ready`
- 随后返回一次 `unread-count`
- 实时推送：
  - `notification.created`
  - `notification.read`
- 每 15 秒发送一次心跳注释帧

后续可扩展：
- `Last-Event-ID`
- 按 `source` / `level` 过滤
- Redis 多实例订阅同步

#### 未读数轮询接口
```http
GET /api/v1/public/notifications/unread-count
```

用于前端 header 红点、角标刷新。

### 12.3 第三方通知接口（后续）

#### Webhook 订阅配置
```http
POST /api/v1/public/notifications/webhooks
PATCH /api/v1/public/notifications/webhooks/:uid
GET /api/v1/public/notifications/webhooks
DELETE /api/v1/public/notifications/webhooks/:uid
```

配置项建议包括：
- `applicationUid`
- `events[]`
- `targetUrl`
- `secret`
- `enabled`

### 12.4 管理接口（后续）

#### 重新投递失败通知
```http
POST /api/v1/admin/notifications/:uid/retry
```

#### 查询投递日志
```http
GET /api/v1/admin/notifications/:uid/deliveries
```

#### 手动发送系统通知
```http
POST /api/v1/admin/notifications/broadcast
```

## 13. 未读/已读模型

站内通知建议使用“事件表 + 收件箱表”双层模型，而不是在通知主表上直接记录已读状态。

原因：
- 一条通知可能面向多个接收人
- 不同接收人的已读状态必须独立
- 后续支持 owner / approver / participant 等多类受众时更容易扩展

推荐规则：
- `notifications` 只保存事件本体
- `notification_inboxes` 保存“用户是否已读”
- 前端未读数以 `notification_inboxes.isRead=false` 聚合

## 14. SSE 模型

通知 SSE 建议复用现有实时事件基础设施，但单独定义通知流语义。

建议事件格式：

```text
event: notification
id: 1745827200000:notification:uid-123
data: {"uid":"uid-123","type":"audit.approved","title":"审核已通过"}
```

建议设计：
- `id` 采用时间戳 + 事件类型 + 业务 uid 的组合，便于断点续传
- 支持服务端按用户身份过滤
- 支持与 Redis Streams 结合，满足多实例部署

## 15. 第一阶段最小实现范围

建议先只做这一组最小能力：

- 通知主表 `notifications`
- 收件箱表 `notification_inboxes`
- 用户列表查询接口
- 单条已读接口
- 未读计数接口
- SSE 实时推送

第一阶段先不做：
- 邮件 / 短信
- Webhook 重试编排
- 模板管理
- 死信队列

这样可以先把“通知中心”从概念域落成真正可交付的产品能力。 

## 15. 待办

当前这一轮先收口到“可用的一期”，剩余事项放到后续迭代：

- 增加独立的通知中心页面，而不只停留在 Header 下拉面板
- 支持通知分组与筛选，例如审批 / 授权 / 系统
- 增加 `GET /api/v1/public/notifications/:uid` 通知详情接口
- 支持 `Last-Event-ID`，完善 SSE 断线续传
- 将通知 SSE 从单实例内存分发升级为多实例一致性分发
- 为 Webhook 增加订阅配置、签名校验、失败重试和投递日志
- 增加应用发布、下架、删除、配置变更等更多业务事件源
- 优化通知文案模板体系，避免事件类型增多后文案分散在业务代码里
- 增加“刚刚处理”之外的更完整交互反馈，例如处理完成后的局部动画与定位
- 评估通知与审批工作台、我的配置页之间的统一跳转协议
