# 运行配置

本文档说明 Node 服务的运行时配置来源、加载优先级、默认值，以及哪些项支持环境变量覆盖。

## 配置加载方式
- 默认从项目根目录的 `config.js` 加载。
- 可通过环境变量 `APP_CONFIG_PATH` 指定其他配置文件路径。
- 支持 `.js` / `.cjs` 模块导出，也支持 JSON 文件。

启动前通常先复制模板：

```bash
cp config.js.template config.js
```

配置加载入口见：
- `src/config/runtime.ts`
- `config.js.template`

## 加密密钥文件（生产推荐）
- 生成加密密钥文件：
  - `node scripts/init-secrets.cjs`
  - 默认输出：`run/secrets.enc.json`
- 启动时加载加密密钥文件：
  - `SECRETS_FILE=run/secrets.enc.json bash scripts/starter.sh start`
- 机制说明：
  - `starter.sh` 检测到 `SECRETS_FILE` 存在后，会提示输入密码（或读取 `SECRETS_PASSWORD_FILE`）。
  - Node 进程启动后在内存中解密密钥，不再注入 `JWT_SECRET/UCAN_ISSUER_PRIVATE_KEY/TOTP_AUTH_TOTP_MASTER_KEY` 环境变量。
  - 支持通过 `NODE_SECRETS_PASSWORD` 提供密码（仅建议自动化场景）。
  - `SECRETS_PASSWORD_FILE` 为启动脚本入参；脚本会转为 `NODE_SECRETS_PASSWORD_FILE` 传递给 Node 进程一次性读取。

## 优先级
默认优先级如下：

1. 环境变量
2. `config.js`
3. 代码内置默认值

注意：
- 不是所有配置项都支持环境变量覆盖。
- `ACTION_SIGNATURE_MAX_AGE_MS`、`ACTION_SIGNATURE_FUTURE_SKEW_MS` 是 env-only，不读取 `config.js`。

## 配置项总览

### `app`
- `app.env`
  - 说明：运行环境标识
  - 默认：`dev`
  - 环境变量：无
- `app.port`
  - 说明：HTTP 服务端口
  - 默认：`8100`
  - 环境变量：`APP_PORT`

### `secrets`
- `secrets.file`
  - 说明：加密密钥文件路径（支持相对路径）
  - 默认：`run/secrets.enc.json`
  - 环境变量：`SECRETS_FILE`（优先级更高）

### `database`
- `database.type`
  - 说明：数据库类型，支持 `better-sqlite3` / `sqlite` / `mysql` / `postgres`
- `database.database`
  - 说明：数据库名；`better-sqlite3` / `sqlite` 场景下为文件路径
- `database.host`
- `database.port`
- `database.username`
- `database.password`
- `database.synchronize`
  - 说明：默认模板的本地 sqlite 快速启动模式可开启；mysql/postgres 场景应保持关闭并走 migration
- `database.logging`
- `database.schema`
  - 说明：Postgres schema 名
- 环境变量覆盖：无

### `auth`
- `auth.jwtSecret`
  - 默认：空（建议留空，使用密钥文件或环境变量注入）
  - 环境变量：`JWT_SECRET`
  - 启动安全要求：不可为空、不可使用默认占位值、长度至少 32 字符
  - 生成示例：
    - `openssl rand -hex 32`
    - `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`
- `auth.accessTtlMs`
  - 默认：`15 * 60 * 1000`
  - 环境变量：`ACCESS_TTL_MS`
- `auth.refreshTtlMs`
  - 默认：`7 * 24 * 60 * 60 * 1000`
  - 环境变量：`REFRESH_TTL_MS`
- `auth.challengeTtlMs`
  - 默认：`5 * 60 * 1000`
  - 环境变量：`AUTH_CHALLENGE_TTL_MS`
- `auth.refreshCookieName`
  - 默认：`refresh_token`
  - 环境变量：`AUTH_REFRESH_COOKIE_NAME`
- `auth.cookieSameSite`
  - 默认：`lax`
  - 环境变量：`COOKIE_SAMESITE`
- `auth.cookieSecure`
  - 默认：`false`
  - 环境变量：`COOKIE_SECURE`

### `idempotency`
- `idempotency.successRetentionDays`
  - 说明：成功响应缓存保留天数
  - 默认：`7`
- `idempotency.failureRetentionDays`
  - 说明：失败响应缓存保留天数
  - 默认：`1`
- `idempotency.pendingTimeoutMs`
  - 说明：`pending` 请求超时清理阈值
  - 默认：`15 * 60 * 1000`
- `idempotency.cleanupIntervalMs`
  - 说明：后台清理任务执行间隔
  - 默认：`15 * 60 * 1000`
- 环境变量覆盖：无

说明：
- 成功响应按 `2xx-3xx` 归类。
- 失败响应按 `4xx-5xx` 归类。
- 若旧配置中仍存在 `idempotency.responseRetentionDays`，当前代码会将其作为 `successRetentionDays` 的回退值，并用于 `failureRetentionDays` 的默认推导；新配置应优先使用 `successRetentionDays` / `failureRetentionDays`。

### 写签名时间窗（env-only）
- `ACTION_SIGNATURE_MAX_AGE_MS`
  - 说明：签名最大有效时长
  - 默认：`5 * 60 * 1000`
- `ACTION_SIGNATURE_FUTURE_SKEW_MS`
  - 说明：允许的未来时间偏移
  - 默认：`30 * 1000`

说明：
- 这两个值只从环境变量读取，不支持 `config.js`。
- 代码位置：`src/auth/actionSignature.ts`

### 前端静态目录（env-only）
- `WEB_DIST_DIR`
  - 说明：Node 服务托管前端静态资源时使用的目录
  - 默认：`<project-root>/web/dist`

说明：
- 目录下需要存在 `index.html`
- 未设置且默认目录不存在时，Node 服务仅提供 API，不托管前端页面

### `ucan`
- `ucan.aud`
  - 默认：`did:web:localhost:8100`
  - 环境变量：`UCAN_AUD`
- `ucan.with`
  - 默认：`app:all:localhost-*`
  - 环境变量：`UCAN_WITH`
- `ucan.can`
  - 默认：`invoke`
  - 环境变量：`UCAN_CAN`

说明：
- `UCAN_AUD` 必须与前端生成 token 时的 audience 完全一致。
- 若未配置 `UCAN_AUD`，默认值会基于 `APP_PORT` / `app.port` 推导。

### `ucanIssuer`
- `ucanIssuer.enabled`
  - 默认：`false`
  - 环境变量：`UCAN_ISSUER_ENABLED`
- `ucanIssuer.mode`
  - 默认：`verify`
  - 可选：`verify` / `issue` / `hybrid`
  - 环境变量：`UCAN_ISSUER_MODE`
- `ucanIssuer.did`
  - 默认：空
  - 环境变量：`UCAN_ISSUER_DID`
- `ucanIssuer.privateKey`
  - 默认：空
  - 环境变量：`UCAN_ISSUER_PRIVATE_KEY`
- `ucanIssuer.sessionTtlMs`
  - 默认：`5 * 60 * 1000`
  - 环境变量：`UCAN_ISSUER_SESSION_TTL_MS`
- `ucanIssuer.tokenTtlMs`
  - 默认：`10 * 60 * 1000`
  - 环境变量：`UCAN_ISSUER_TOKEN_TTL_MS`
- `ucanIssuer.defaultAudience`
  - 默认：`ucan.aud`
  - 环境变量：`UCAN_ISSUER_DEFAULT_AUDIENCE`
- `ucanIssuer.defaultCapabilities`
  - 默认：`[{ with: ucan.with, can: ucan.can }]`
  - 环境变量：`UCAN_ISSUER_DEFAULT_CAPABILITIES`（JSON 字符串）

说明：
- `mode=verify`：仅钱包 UCAN 校验，不启用中心化签发。
- `mode=issue`：启用中心化签发，并仅接受中心化 Issuer UCAN。
- `mode=hybrid`：同时接受钱包 UCAN 和中心化 Issuer UCAN。
- 签发模式通过 `/api/v1/public/auth/central/*` 提供 issuer/session/issue/revoke 接口。
- 当 `mode=issue|hybrid` 时，必须配置可用的 `ucanIssuer.privateKey`；若同时配置 `ucanIssuer.did`，必须与私钥推导 DID 一致。
- 当 `ucanIssuer.enabled=true` 且 `mode=issue|hybrid` 时，若签发配置未就绪，服务会在启动前直接失败（安全前置检查）。

### `totpAuth`
- `totpAuth.enabled`
  - 默认：`false`
  - 环境变量：`TOTP_AUTH_ENABLED`
- `totpAuth.issuerName`
  - 默认：`YeYing Node`
  - 环境变量：`TOTP_AUTH_ISSUER_NAME`
- `totpAuth.verifyPath`
  - 默认：`/totp-auth`
  - 环境变量：`TOTP_AUTH_VERIFY_PATH`
- `totpAuth.portalBaseUrl`
  - 默认：`http://127.0.0.1:<app.port>`
  - 环境变量：`TOTP_AUTH_PORTAL_BASE_URL`
- `totpAuth.requestTtlMs`
  - 默认：`2 * 60 * 1000`
  - 环境变量：`TOTP_AUTH_REQUEST_TTL_MS`
- `totpAuth.exchangeCodeTtlMs`
  - 默认：`60 * 1000`
  - 环境变量：`TOTP_AUTH_EXCHANGE_CODE_TTL_MS`
- `totpAuth.codeDigits`
  - 默认：`6`
  - 环境变量：`TOTP_AUTH_CODE_DIGITS`
- `totpAuth.codePeriodSec`
  - 默认：`30`
  - 环境变量：`TOTP_AUTH_CODE_PERIOD_SEC`
- `totpAuth.codeWindow`
  - 默认：`1`
  - 环境变量：`TOTP_AUTH_CODE_WINDOW`
- `totpAuth.maxAttempts`
  - 默认：`5`
  - 环境变量：`TOTP_AUTH_MAX_ATTEMPTS`
- `totpAuth.totpMasterKey`
  - 默认：空
  - 环境变量：`TOTP_AUTH_TOTP_MASTER_KEY`
  - 说明：用于加密存储“按地址生成的随机 TOTP 密钥”，不再使用“主密钥+地址确定性派生”

说明：
- 该配置用于“PC 钱包登录发起 + 手机认证器验证码确认”的中心化 UCAN 登录桥接。
- 对于“手机地址发起登录 + 跳转 Node 输入 TOTP”的场景，`appId` 必须使用应用市场已发布的 `AppId`（`applications.uid`）。
- 回跳地址从应用发布字段 `redirectUris` 读取（仅允许单值），`authorize/request` 与 `authorize/exchange` 中的 `redirectUri` 必须严格命中该字段。
- `authorize/request` 的 `audience/capabilities/appName` 不再由客户端传入决定，统一按服务端应用策略解析。
- 应用必须在发布信息中配置中心化签发策略：
  - `ucanAudience`：签发 UCAN 的 `aud`
  - `ucanCapabilities`：签发 UCAN 的能力列表
  - 缺失会返回 `TOTP_AUTH_APP_POLICY_NOT_CONFIGURED`
- `authorize/request`、`authorize/approve`、`authorize/exchange` 已增加服务端限流。
- 当 `totpAuth.enabled=true` 时，必须配置 `totpAuth.totpMasterKey`。
- 当 `totpAuth.enabled=true` 且主密钥不可用时，服务会在启动前直接失败（安全前置检查）。
- 若 `totpAuth` 未启用，不影响现有 SIWE/JWT 和 `/auth/central/*` 路径。

### `logger`
- `logger.level`
  - 默认：`info`

  - 默认：`true`
  - 说明：是否保留控制台输出

- `maxFiles=14d`

## 数据库迁移与缺列排查

Postgres 模式下，服务启动会自动执行迁移；本次中心化 UCAN 改造新增了应用回跳字段迁移：

- `20260423121000-add-application-redirect-uris.ts`
  - 变更：`applications.redirect_uris`（`text NOT NULL DEFAULT ''`）
- `20260423182000-add-totp-subject-secrets.ts`
  - 变更：`totp_subject_secrets`（按地址存储加密后的随机 TOTP 密钥与绑定状态）
- `20260423193000-add-application-ucan-policy.ts`
  - 变更：
    - `applications.ucan_audience`（`text NOT NULL DEFAULT ''`）
    - `applications.ucan_capabilities`（`text NOT NULL DEFAULT ''`）

如遇到以下错误：

- `Search applications failed: column ApplicationDO.redirect_uris does not exist`

请先确认服务已重启并执行到最新迁移。若是历史库需要临时补齐，可手动执行：

```sql
ALTER TABLE node.applications
ADD COLUMN IF NOT EXISTS redirect_uris text NOT NULL DEFAULT '';
```

说明：
- `node` 为默认 schema，若你的环境使用 `public` 或其他 schema，请替换为实际 schema。
- 补列后建议重启 Node 服务，确保实体字段与数据库结构一致。

## 推荐最小配置
至少应明确以下几项：
- `database.*`
- `auth.jwtSecret`（或启用密文方案并提供 `secrets.file` + 口令）
- `ucan.aud`
- `audit.approvers`
- `audit.requiredApprovals`

生产前建议额外明确：
- `auth.cookieSecure=true`
- `logger.file.*`
- `redis.*`（若需要 MPC 跨实例）
